# Framework

The `@xyd-js/framework` package is the React runtime layer that connects configuration, navigation, and page data to the UI. It provides context providers, hooks, pre-built components, and server-side utilities used by themes and plugins to render documentation pages.

## Package Structure

The package is split into three entry points:

| Export | Entry | Purpose |
|--------|-------|---------|
| `@xyd-js/framework` | `src/index.ts` | Core utilities: `pageMetaLayout`, `Surfaces` |
| `@xyd-js/framework/react` | `packages/react/index.ts` | React contexts, hooks, and components |
| `@xyd-js/framework/hydration` | `packages/hydration/index.ts` | Server-side `mapSettingsToProps` |

## Two-Tier Context System

The framework uses two nested React contexts to separate app-wide state from page-specific state.

### FrameworkContext (app-wide)

Wraps the entire application. Provides settings, sidebar groups, metadata, component overrides, and banner content. All values are frozen with `Object.freeze` to prevent accidental mutation.

| Provided Data | Type | Description |
|---------------|------|-------------|
| `settings` | `Readonly<Settings>` | Complete processed configuration |
| `sidebarGroups` | `Readonly<FwSidebarItemProps[]>` | Sidebar navigation tree |
| `metadata` | `Readonly<Metadata>` | Current page metadata |
| `components` | `Readonly<ComponentMap>` | Component overrides from plugins |
| `BannerContent` | `React.ComponentType \| null` | Banner component if configured |

### FrameworkPageContext (page-specific)

Wraps individual page content inside `FrameworkContext`. Provides compiled MDX content, TOC, breadcrumbs, nav links, and edit link for the current page.

| Provided Data | Type | Description |
|---------------|------|-------------|
| `ContentComponent` | `(props) => JSX.Element` | Compiled MDX with full component set |
| `ContentOriginal` | `(props) => JSX.Element` | Compiled MDX without file components |
| `metadata` | `Readonly<Metadata>` | Page frontmatter |
| `toc` | `Readonly<ITOC[]>` | Table of contents entries |
| `breadcrumbs` | `Readonly<IBreadcrumb[]>` | Breadcrumb path to current page |
| `navlinks` | `Readonly<INavLinks>` | Previous/next page links |
| `editLink` | `Readonly<string>` | GitHub edit URL |
| `rawPage` | `Readonly<string>` | Raw markdown source |

## Hooks

### App-Level Hooks (from FrameworkContext)

| Hook | Returns | Description |
|------|---------|-------------|
| `useSettings()` | `Settings` | Complete site configuration |
| `useMetadata()` | `Metadata` | Current page metadata |
| `useComponents()` | `ComponentMap` | Plugin component overrides |
| `useAppearance()` | `Appearance` | Theme appearance settings |
| `useShowColorSchemeButton()` | `boolean` | Whether to show light/dark toggle |
| `useBannerContent()` | `ComponentType \| null` | Banner component |
| `useSidebarGroups()` | `FwSidebarItemProps[]` | Sidebar navigation tree |

### Page-Level Hooks (from FrameworkPageContext)

| Hook | Returns | Description |
|------|---------|-------------|
| `useContentComponent()` | `Component` | Compiled MDX with all components |
| `useContentOriginal()` | `Component` | Compiled MDX without file-scoped components |
| `useToC()` | `ITOC[]` | Table of contents |
| `useBreadcrumbs()` | `IBreadcrumb[]` | Breadcrumb path |
| `useNavLinks()` | `INavLinks` | Previous/next page links |
| `useEditLink()` | `string` | GitHub edit URL |
| `useRawPage()` | `string` | Raw markdown source |

## Surfaces System

Surfaces provide named injection points where themes and plugins can place UI elements without modifying layout code directly.

| Surface Target | Location |
|----------------|----------|
| `nav.right` | Right side of the navigation bar |
| `sidebar.top` | Top of the sidebar |
| `sidebar.item.left` | Left side of sidebar items |
| `sidebar.item.right` | Right side of sidebar items |
| `page.footer.bottom` | Bottom of the page footer |

Themes call `surfaces.define(target, component)` to register elements. Components read them via `surfaces.get(target)` through the `SurfaceContext`.

## Server-Side: mapSettingsToProps

The `mapSettingsToProps` function (exported from `@xyd-js/framework/hydration`) runs server-side in the page loader. It transforms raw `Settings` and a URL slug into the props needed by the Framework contexts:

1. **Filter navigation** — finds the sidebar section matching the current route (supports route-based multi-sidebar)
2. **Load frontmatters** — reads title, icon, and metadata from all pages in the filtered navigation
3. **Build sidebar groups** — maps pages to `FwSidebarItemProps` tree, resolving groups, icons, and hidden pages
4. **Build breadcrumbs** — traces the path from root to the current page through the navigation hierarchy
5. **Build nav links** — flattens all pages and finds previous/next neighbors, handling nested groups via BFS

Returns `{ groups, breadcrumbs, navlinks, hiddenPages, metadata }`.

## Page Meta Layout

The `pageMetaLayout` function determines the layout mode based on page metadata. Pages with `component` set to `home`, `firstslide`, or `bloghome` use the `"page"` layout (full-width, no sidebar). All other pages use the default documentation layout.

## Integration with Themes

Themes extend `BaseTheme` and consume framework hooks in their `Layout`, `Page`, `Content`, and other methods. The typical render flow:

1. `Layout` reads `useSettings()`, `useAppearance()`, `useSidebarGroups()` to build the shell
2. `Page` reads `useContentComponent()`, `useToC()`, `useBreadcrumbs()`, `useNavLinks()` to render content
3. Framework components (`FwSidebar`, `FwNav`, `FwToc`, etc.) handle the actual rendering with data from hooks
4. Surfaces allow themes to inject additional UI elements at defined points
