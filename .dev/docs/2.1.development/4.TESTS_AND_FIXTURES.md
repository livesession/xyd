# Tests and Fixtures

## Directory layout

Tests live in `__tests__/` and fixtures in `__fixtures__/` at the package root (or under `src/` for some packages).

```
packages/xyd-<name>/
├── __tests__/
│   ├── <feature>.test.ts    # Test file
│   └── utils.ts             # Shared test helpers
└── __fixtures__/
    ├── 1.basic/             # Positive fixture (runs in CI)
    ├── 2.circular/
    ├── -1.opendocs.flat/    # Negative-numbered (advanced/optional, may be disabled)
    └── -2.complex.openai/
```

## Fixture naming convention

Each fixture is a numbered directory: `<order>.<descriptive-name>`.

| Prefix | Meaning | Example |
|--------|---------|---------|
| `1.`, `2.`, `3.` ... | Standard test cases, run in CI | `1.basic`, `3.multiple-responses` |
| `-1.`, `-2.`, `-3.` ... | Advanced/optional cases, may be commented out or sorted first | `-1.opendocs.flat`, `-2.complex.openai` |

Use dots to add sub-categories: `5.xdocs.sidebar`, `-1.opendocs.sort+group`.

## Input / output pattern

Every fixture contains an **input** and an expected **output**. The format depends on what's being tested:

| Package | Input | Output |
|---------|-------|--------|
| `xyd-gql` | `input.graphql` | `output.json` |
| `xyd-openapi` | `input.yaml` | `output.json` |
| `xyd-opencli-remark` | `input.md` | `output.md` |
| `xyd-source-react-runtime` | `input/` (self-contained app with `src/`, `package.json`, build config) | `output.js` |

For simple converters (OpenAPI, GraphQL, remark), input and output are single files. For build plugins like `xyd-source-react-runtime`, each fixture is a self-contained app with its own bundler config (`vite.config.ts`, `rollup.config.mjs`, `esbuild.config.mjs`) and a `build` script in `package.json`. The test runs `pnpm build` and snapshots the output.

Fixture naming can also encode the build environment: `1.vite-lib.user-card`, `5.rollup.user-card`, `7.react-router.app`.

## Test structure

Tests are data-driven. A test file defines an array of cases and iterates through them using a shared utility:

```ts
// __tests__/<feature>.test.ts
import { testMyFeature } from './utils';

const tests = [
    { name: '1.basic', description: 'Basic example' },
    { name: '2.more',  description: 'Extended example' },
];

describe('my-feature', () => {
    for (const t of tests) {
        it(t.description, async () => {
            await testMyFeature(t.name);
        });
    }
});
```

The utility function in `utils.ts` handles:
1. Loading input from `__fixtures__/<name>/`
2. Running the transformation
3. Saving result to `output.*` (for snapshot regeneration)
4. Comparing result against expected `output.*`

## Running tests

```bash
# Run all unit tests
pnpm test:unit

# Run tests for a specific package
cd packages/xyd-<name>
pnpm test

# Run with vitest watch mode
pnpm vitest
```

## E2E

E2E tests live in `__tests__/e2e/` and use Playwright with Chromium. Each test starts an xyd server (dev or build mode), navigates pages, and asserts rendered output.

### Directory structure

```
__tests__/e2e/
├── 1.navigation/
├── 2.openapi/
├── 3.graphql/
├── 4.writing/
├── 5.access-control/
├── 6.custom-vite-options/
├── 7.themes/
│   ├── 1.dynamic-theme-resolution/
│   └── 2.custom-npm-theme/
└── utils/
    └── xyd-server.ts        # Shared server utility
```

Each test directory contains:
- A `.test.ts` file with the Playwright test suite
- A `docs.json` configuration
- Content files (markdown)
- Optional `package.json` for dependency declarations

### XydServer utility

The `XydServer` class (`__tests__/e2e/utils/xyd-server.ts`) handles spinning up xyd for tests:

| Mode | What it does |
|------|---|
| `"dev"` | Starts `xyd dev` with HMR |
| `"build"` | Runs `xyd build`, then serves static output |

CLI resolution priority:
1. `XYD_LOCAL_TEST_VERSION` — installs specific version to isolated `/tmp` dir
2. Monorepo local CLI at `packages/xyd-cli/dist/index.js` with `XYD_DEV_MODE=1`
3. Global `xyd` command

### Running e2e tests locally

```bash
# All e2e tests (uses monorepo local CLI)
pnpm run test:e2e

# Specific test
pnpm run test:e2e __tests__/e2e/7.themes

# With Playwright UI
pnpm run test:e2e:ui
```

### Running e2e tests in Docker

Docker provides an isolated environment with Verdaccio (local npm registry). This lets you test with a freshly-built xyd version published to Verdaccio, exactly like a real user would install from npm.

#### Setup

```
__tests__/docker/
├── Dockerfile              # Base image: Node 22, pnpm, bun, Chromium, Verdaccio
├── entrypoint.sh           # Versioning → rebuild → publish → test
├── run-e2e.sh              # Local build + Docker run with timing
└── verdaccio-config.yaml   # Local registry config
```

#### Flow

1. **Local** (`run-e2e.sh`): builds all packages on your machine
2. **Docker** (`entrypoint.sh`): mounts repo as volume, then:
   - Starts Verdaccio
   - Versions packages with changeset snapshot
   - Rebuilds `xyd-documan` (bakes new theme versions)
   - Publishes all `@xyd-js/*` packages to Verdaccio
   - Publishes `xyd-js` CLI wrapper
   - Pre-installs xyd-js to isolated dir (for `XydServer`)
   - Runs Playwright tests

#### Commands

```bash
# First run (builds base Docker image + runs tests)
./__tests__/docker/run-e2e.sh -- __tests__/e2e/7.themes

# Subsequent runs (reuses cached base image)
./__tests__/docker/run-e2e.sh -- __tests__/e2e/7.themes/2.custom-npm-theme

# Rebuild base image (after Dockerfile changes)
./__tests__/docker/run-e2e.sh --rebuild-base -- __tests__/e2e/7.themes

# Save output with timestamps
./__tests__/docker/run-e2e.sh -- __tests__/e2e/7.themes 2>&1 | tee e2e-output.log
```

#### Timing (typical run)

> e2e tests inside Docker are slow - we need to improve that in the future.
 
| Phase | Time |
|---|---|
| Local build | ~100s |
| Verdaccio + versioning + documan rebuild | ~50s |
| Publishing to Verdaccio | ~45s |
| Pre-install xyd-js | ~5s |
| Running tests | ~150s |

#### Tests requiring Verdaccio

Some tests publish packages to a local registry (e.g., custom npm themes). In Docker, the entrypoint's Verdaccio is reused via `npm_config_registry` env var. In CI, a Verdaccio step is added to the workflow.

Example: `7.themes/2.custom-npm-theme` publishes a fixture theme package to Verdaccio before running `xyd build`.
