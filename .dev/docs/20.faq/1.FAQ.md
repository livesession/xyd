# FAQ

## Getting Started

**Q: What Node.js version do I need?**
A: Node.js >= 22.12.0. Tested on versions 22, 23, and 24.

**Q: Which package managers are supported?**
A: npm, pnpm, bun, yarn. The CLI auto-detects your package manager. For pnpm, you may need `XYD_NODE_PM=pnpm`.

**Q: How do I create a new docs site from scratch?**
A: Install `xyd-js` globally, create a `docs.json` and a `content/` directory with markdown files, then run `xyd` to start the dev server. See `1.getting-started/CreatingYourFirstSite.md`.

**Q: How do I deploy my site?**
A: Run `xyd build` and deploy `.xyd/build/client/` to any static hosting (Netlify, Vercel, GitHub Pages, CloudFlare). No server runtime needed.

## Configuration

**Q: Can I use TypeScript for configuration?**
A: `docs.ts` / `docs.tsx` is experimentally supported but not fully stable yet. The recommended format is `docs.json`.

**Q: Can I use environment variables in my config?**
A: Yes. Use `$VAR_NAME` syntax in `docs.json`. Variables are loaded from `.env`, `.env.local`, `.env.development`, `.env.production` files.

**Q: How do I set a base path for my site (e.g., `/docs`)?**
A: Set `advanced.basename` in your config: `{ "advanced": { "basename": "/docs" } }`.

**Q: Can I customize the Vite configuration?**
A: Yes. Use `advanced.vite` in your config to merge custom Vite settings (server port, allowed hosts, aliases, etc.).

## Themes and Customization

**Q: Can I customize theme colors without creating a custom theme?**
A: Yes. Use `theme.appearance` to set primary colors, color scheme (light/dark/auto), CSS tokens, and fonts.

**Q: Can I use custom fonts?**
A: Yes. Configure `theme.fonts` with font file paths (.woff2, .woff, .ttf). Automatic `@font-face` generation.

**Q: Can I use custom icons?**
A: Yes. Set `theme.icons.library` to use Iconify icon sets.

**Q: Can I create a custom theme?**
A: Yes. Extend the `BaseTheme` class and implement `Layout` and `Page` methods. See `10.themes/ThemeArchitecture.md`.

## Content

**Q: What content formats are supported?**
A: Markdown (`.md`) and MDX (`.mdx`). MDX allows embedding React components in content.

**Q: Can I use React components in my markdown?**
A: Yes, use `.mdx` files. Built-in components include Callout, Tabs, Steps, Badge, Card, and code groups.

**Q: How do I add diagrams?**
A: Set `integrations.diagrams` to `true` (defaults to Mermaid) or `["mermaid", "graphviz"]` for both.

**Q: How do I add math equations?**
A: Math is supported out of the box via KaTeX. Use `$` for inline and `$$` for block math.

**Q: What is frontmatter?**
A: YAML block at the top of markdown files (`---` delimited). Use `title`, `description`, `icon`, `seoTitle`, etc.

**Q: Can I compose auto-generated API docs with manual content?**
A: Yes. Set `uniform:` in frontmatter to point to an API spec or source file. Content below the frontmatter is merged with the generated documentation. See `16.composer/1.COMPOSER.md`.

## API Documentation

**Q: Which API spec formats are supported?**
A: OpenAPI 3.x (YAML/JSON), GraphQL (SDL), and TypeScript/React source code.

**Q: How do I add OpenAPI documentation?**
A: Set `uniform: "./path/to/openapi.yaml"` in page frontmatter, or configure `api.openapi` in settings for bulk processing.

**Q: How do I add GraphQL documentation?**
A: Set `graphql: "./path/to/schema.graphql"` in page frontmatter or configure `api.graphql` in settings.

**Q: Can I generate docs from React component source code?**
A: Yes. Set `uniform: "@components/MyComponent.tsx"` in frontmatter. Uses TypeDoc for type extraction.

**Q: Are code samples auto-generated?**
A: Yes, for OpenAPI specs. Generates samples in shell, JavaScript, Python, and Go by default. Custom languages via `x-docs.codeLanguages` extension.

## Plugins

**Q: How do I create a custom plugin?**
A: Export a factory function that receives settings and returns a `PluginConfig` with components, markdown plugins, hooks, etc. See `12.plugins-and-integrations/CreatingCustomPlugins.md`.

**Q: How do I add a plugin to my site?**
A: Add the plugin package name to the `plugins` array in `docs.json`: `{ "plugins": ["@company/my-plugin"] }`.

**Q: What can a plugin contribute?**
A: Vite plugins, React components, remark/rehype markdown plugins, lifecycle hooks, and HTML head elements.

**Q: What integrations are available out of the box?**
A: Analytics (LiveSession), search (Orama, Algolia), support (Chatwoot, Intercom, LiveChat), diagrams (Mermaid, Graphviz).

## Navigation

**Q: How do I configure the sidebar?**
A: Use `navigation.sidebar` in settings. Supports hierarchical groups, route-scoped sections, and page ordering.

**Q: Can I have multiple sidebars for different sections?**
A: Yes. Use `SidebarRoute` objects with a `route` field to scope sidebar content to specific URL prefixes.

**Q: How do I add tabs?**
A: Use `navigation.tabs` in settings. Each tab has a `page` (route matching) and `href` (navigation target).

**Q: How do I add links to the header (GitHub, social, etc.)?**
A: Use `navigation.anchors.header`. Supports regular links, buttons (`button: "primary"`), and social links (`social: "github"`).

## Access Control

**Q: Can I protect pages with authentication?**
A: Yes. Configure `accessControl` in `docs.json` with JWT or OAuth provider. No manual plugin installation needed.

**Q: Does access control work on static hosting?**
A: Layer 1 (SSR exclusion) works on any static host — protected content is code-split and loaded after auth. Layer 2 (deploy adapters) adds server-side verification for Node.js, Netlify, Vercel, or Cloudflare.

**Q: Can I restrict pages to specific user groups?**
A: Yes. Use `accessControl.rules` with `groups` field, or set `accessGroups` in page frontmatter.

## SEO and AI

**Q: How does SEO work?**
A: Frontmatter `title` and `description` are used for meta tags. Override with `seoTitle` and `seoDescription`. Configure `seo.domain` for canonical URLs.

**Q: What is llms.txt?**
A: An auto-generated file for AI indexing, built from page frontmatter. Configure via `ai.llmsTxt`. Protected pages are excluded automatically.

## Development

**Q: How do I set up the monorepo for development?**
A: Clone the repo, run `pnpm install`, then `pnpm dev` for watch mode builds. Requires Node.js >= 22.12.0 and pnpm >= 9.

**Q: How do I run tests?**
A: `pnpm test:unit` for Vitest unit tests, `pnpm test:e2e` for Playwright E2E tests, `pnpm test:node-support` for Node.js compatibility matrix.

**Q: How do I test my package changes locally?**
A: Use Verdaccio as a local npm registry. Run `verdaccio`, publish with `npm_config_registry=http://localhost:4873 pnpm changeset publish`, then install from it.

**Q: How do fixtures work?**
A: Each fixture is a numbered directory under `__fixtures__/` with `input.*` and `output.*` files. Tests are data-driven — iterate through fixtures and compare results. See `2.1.development/1.DEVELOPMENT.md`.
